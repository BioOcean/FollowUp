## 📋 重构流程（四步法）

### 第一步：分析 NTCare 旧系统实现
1. **定位旧系统页面**：在 `F:\project\NTCare` 中找到我指定的页面对应的源代码文件
2. **功能清单梳理**：列出该页面的**所有功能点**（左侧菜单栏的导航功能除外，这些会在后续统一实现）
3. **技术实现分析**：针对每个功能点，找出：
   - 业务逻辑实现代码
   - 调用的后端接口（API endpoints）
   - 涉及的数据库表及字段
   - 关键的数据流转逻辑

### 第二步：设计新系统重构方案
1. **参考架构规范**：严格遵循 `@CLAUDE.md` 中定义的模块化架构规范
2. **数据库适配**：
   - 旧系统数据库通过 `postgres` MCP 访问
   - 新系统数据库通过 `postgres-new` MCP 访问
   - **必须参考** `@f:\project\FollowUp\REFACTORING_SQL.md` 了解数据库结构差异
   - 使用新数据库的表结构和字段重新设计数据访问逻辑
3. **UI 设计**：参考 `@f:\project\FollowUp\docs\UI-Design-Guidelines.md` 重新设计界面布局和交互
4. **输出重构方案**，包含：
   - 文件结构（Pages/Components/Dialogs/Models/Services）
   - 每个文件的职责说明
   - 数据库查询逻辑（使用新表结构）
   - UI 布局设计
   - 与旧系统的差异对比

### 第三步：方案审查与修改
1. **提交方案**：将完整的重构方案提交给我审查
2. **等待确认**：我会提出修改意见或确认方案可行
3. **迭代优化**：根据反馈修改方案，直到我明确表示"可以实施"

### 第四步：实施重构
1. **仅在我明确确认后**才开始编写代码
2. 严格按照审查通过的方案实施
3. 实施过程中如遇问题，及时反馈并讨论调整方案

## ⚠️ 关键注意事项
- **功能完整性**：
  - 重构时严禁丢失任何现有功能和数据展示
  - 重构时严禁随意添加未经确认的新功能
  - 重构前必须详细对比新旧功能清单，确保 100% 功能覆盖
- **数据库差异**：新旧系统数据库结构不同，必须查阅 `REFACTORING_SQL.md` 确认字段映射关系
- **MCP 工具使用**：
  - 查询旧系统数据库 → 使用 `query_postgres`
  - 查询新系统数据库 → 使用 `query_postgres-new`
- **UI 规范**：必须遵循 `UI-Design-Guidelines.md` 的设计原则
- **架构规范**：必须遵循 `CLAUDE.md`（即极简开发规则）的模块化结构
